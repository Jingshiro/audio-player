## 一、安全性修复（5 项）

### S2 — 所有 API 路由添加认证保护

**背景**：当前仅 `settings.js` 使用了 `requireAuth` 中间件，其余路由任何人可直接访问。

**改法**：在以下 6 个路由文件顶部（`const router = express.Router()` 之后）各加一行：

```js
const { requireAuth } = require('../middleware/auth')
router.use(requireAuth)
```

| 文件 | 修改内容 |
|------|----------|
| `backend/src/routes/audio.js` | 添加 requireAuth 导入 + router.use |
| `backend/src/routes/lyrics.js` | 同上 |
| `backend/src/routes/folder.js` | 同上 |
| `backend/src/routes/prompt.js` | 同上 |
| `backend/src/routes/ai.js` | 同上 |
| `backend/src/routes/storage.js` | 同上 |

**不动的路由**：`auth.js` 的 `POST /login` 和 `GET /check` 保持公开（它们本身不需要认证）。

**用户体验影响**：无。用户登录后 token 存在 localStorage，后续请求自动带 `Authorization: Bearer xxx` 头，不会重复要求登录。

### S3 — 密码哈希从 SHA-256 升级为 bcrypt

**背景**：当前使用无盐 SHA-256，易受彩虹表攻击。

**改法**：

1. 安装依赖：`npm install bcryptjs`（纯 JS 实现，无需编译）
2. 修改 `backend/src/routes/auth.js`：

```js
// 替换前
const crypto = require('crypto')
function hashPassword(password) {
  return crypto.createHash('sha256').update(password).digest('hex')
}

// 替换后
const bcrypt = require('bcryptjs')
const SALT_ROUNDS = 10
function hashPassword(password) {
  return bcrypt.hashSync(password, SALT_ROUNDS)
}
function comparePassword(password, hash) {
  return bcrypt.compareSync(password, hash)
}
```

3. 修改登录验证（第 50 行）：

```js
// 替换前
if (!stored || !safeCompare(stored.value, hashPassword(password))) {

// 替换后
if (!stored || !comparePassword(password, stored.value)) {
```

4. 修改密码初始化（第 28 行）：保持不变，`hashPassword` 内部已改用 bcrypt
5. 修改密码更新（第 109 行）：保持不变，`hashPassword(newPassword)` 内部已改用 bcrypt

**兼容性**：首次启动时，旧的 SHA-256 哈希无法通过 bcrypt 验证，用户需要重新输入默认密码登录后修改密码。建议在控制台输出提示。

**已部署项目**：需要手动删除数据库中的 `auth_password` 记录，或删除整个数据库重建，让系统重新初始化默认密码。

### S4 — 数据库文件移出静态目录

**背景**：当前数据库 `music.db` 存在 `./data/` 目录下，而 `express.static(uploadDir)` 将整个 `./data/` 暴露为静态文件，任何人可直接下载 `/data/music.db`。

**改法**：

1. 修改 `backend/src/db/init.js`，将默认数据库路径从 `./data/music.db` 改为 `./db/music.db`：

```js
// 替换前
let dbPath = process.env.DATABASE_URL || './data/music.db'

// 替换后
let dbPath = process.env.DATABASE_URL || './db/music.db'
```

2. 修改 `nginx/default.conf`，关闭目录列表：

```nginx
# 替换前
location /data/ {
    alias /usr/share/nginx/html/data/;
    autoindex on;
}

# 替换后
location /data/ {
    alias /usr/share/nginx/html/data/;
    autoindex off;
}
```

3. 修改 `.gitignore`，添加 `db/` 目录（如果还没有的话）

**已部署项目操作步骤**：
1. 停止服务 `docker compose down`
2. 复制数据库：`cp data/music.db db/music.db`（需先创建 `db/` 目录）
3. 更新代码和 nginx 配置
4. 重启服务 `docker compose up -d`
5. 确认无误后删除旧的 `data/music.db`

### S5 — 密码修改端点修复时序攻击

**背景**：`auth.js` 第 105 行密码修改使用 `!==` 直接比较字符串，存在时序攻击风险。同文件的登录端点已正确使用 `safeCompare`。

**改法**：修改 `backend/src/routes/auth.js` 第 105 行：

```js
// 替换前
if (!stored || stored.value !== hashPassword(oldPassword)) {

// 替换后（如果用 bcrypt）
if (!stored || !comparePassword(oldPassword, stored.value)) {
```

如果采用 S3 的 bcrypt 方案，这里直接用 `comparePassword` 即可，bcrypt 内部已做常量时间比较。

### S9 — AI 代理日志脱敏

**背景**：`ai.js` 的 STT 端点将 `baseUrl`（Gemini 格式下包含 API Key）输出到 console.log。

**改法**：修改 `backend/src/routes/ai.js` 第 91-96 行：

```js
// 替换前
console.log('[AI/STT] baseUrl:', baseUrl)
console.log('[AI/STT] model:', model)

// 替换后
console.log('[AI/STT] baseUrl:', baseUrl ? new URL(baseUrl).hostname : 'N/A')
```

只记录域名，不记录完整 URL（避免泄露 query 参数中的 API Key）。

## 二、易用性修复（5 项）

### U1 — 修复音频列表搜索功能

**背景**：`LibraryView.vue` 的 `onSearch()` 回调体为空，搜索框对音频 tab 无效。台词 tab 的搜索通过 `filteredSubtitles` 计算属性正常工作。

**改法**：不需要修改 `onSearch()` 本身（它本来就是空的防抖壳）。问题在于 `unifiedLibraryStore` 已经有了 `searchQuery` 和 `filteredFiles` 的搜索逻辑，但 `LibraryView` 的搜索框 `v-model` 绑定的是本地 `searchQuery`，没有同步到 store。

修改 `frontend/src/views/LibraryView.vue`：

1. 在 `onSearch()` 中添加同步到 store 的逻辑：

```js
function onSearch() {
  clearTimeout(searchTimer)
  searchTimer = setTimeout(() => {
    unifiedLibraryStore.setSearchQuery(searchQuery.value)
  }, 200)
}
```

2. 同时在 tab 切换时重置搜索（如果需要）：

```js
// 切换 tab 时，搜索框的值需要同步到对应 store
watch(activeTab, () => {
  searchQuery.value = ''
  unifiedLibraryStore.setSearchQuery('')
})
```

这样音频 tab 的搜索就会通过 `unifiedLibraryStore.filteredFiles` 生效。

### U4 — 新增列表循环/列表随机播放模式

**背景**：当前播放列表播到最后一首就停止，没有循环和随机模式。

**改法**：修改 `frontend/src/stores/player.js`：

1. 新增状态：

```js
const loopMode = ref('none') // 'none' | 'list' | 'single' | 'shuffle'
```

2. 新增切换方法：

```js
function toggleLoopMode() {
  const modes = ['none', 'list', 'single', 'shuffle']
  const idx = modes.indexOf(loopMode.value)
  loopMode.value = modes[(idx + 1) % modes.length]
}
```

3. 修改 `playNext()`：

```js
function playNext() {
  if (playlist.value.length === 0) return

  if (loopMode.value === 'single') {
    // 单曲循环：重新播放当前曲目
    loadTrack(playlist.value[currentIndex.value])
    play()
    return
  }

  let nextIndex = currentIndex.value + 1

  if (loopMode.value === 'shuffle') {
    // 随机：随机选一首（不重复当前）
    if (playlist.value.length > 1) {
      do { nextIndex = Math.floor(Math.random() * playlist.value.length) }
      while (nextIndex === currentIndex.value)
    } else {
      nextIndex = 0
    }
  } else if (nextIndex >= playlist.value.length) {
    if (loopMode.value === 'list') {
      nextIndex = 0 // 列表循环：回到第一首
    } else {
      return // 不循环：停止
    }
  }

  currentIndex.value = nextIndex
  loadTrack(playlist.value[nextIndex])
  play()
}
```

4. 修改 `playPrev()`：类似逻辑，支持循环回绕

5. 在 return 中导出 `loopMode` 和 `toggleLoopMode`

6. 修改 `ended` 事件处理（第 143 行）：保持 `playNext()` 调用不变，因为 `playNext` 内部已处理循环逻辑

**UI 部分**：在播放器控制栏添加循环模式切换按钮，显示当前模式图标。

**关于合并本地+服务器音频到播放列表**：`unifiedLibraryStore.audioFiles` 已经合并了本地和服务器音频。播放时通过 `unifiedLibraryStore.getAudioWithUrl(id)` 获取播放 URL（本地用 IndexedDB blob URL，服务器用 `/api/audio/{id}/stream`）。需要在用户点击音频时将其加入 playlist 并设置正确的 url。这个逻辑需要在 PlayerView 或 LibraryView 的点击播放事件中处理。

### U18 — PlayerView 合并重复的 onMounted

**背景**：`PlayerView.vue` 有两个 `onMounted`，第一个加载数据，第二个注册键盘事件。第一个中的 `unifiedLibraryStore.loadAll()` 和 `unifiedSubtitlesStore.loadAll()` 与 `App.vue` 的 `onMounted` 重复。

**改法**：修改 `frontend/src/views/PlayerView.vue`：

1. 合并两个 `onMounted` 为一个
2. 移除重复的 `loadAll()` 调用（App.vue 已加载）

```js
// 替换前（两个 onMounted）
onMounted(async () => {
  const audioEl = document.getElementById('global-audio')
  if (audioEl) {
    playerStore.setAudioElement(audioEl)
  }
  await unifiedLibraryStore.loadAll()
  await unifiedSubtitlesStore.loadAll()
})

// ... 中间其他代码 ...

onMounted(() => {
  document.addEventListener('keydown', handleKeydown)
})

// 替换后（合并为一个）
onMounted(() => {
  const audioEl = document.getElementById('global-audio')
  if (audioEl) {
    playerStore.setAudioElement(audioEl)
  }
  document.addEventListener('keydown', handleKeydown)
})
```

注意：移除 `loadAll()` 后需要确保 App.vue 的 `onMounted` 先于 PlayerView 执行。由于 Vue 路由机制，App.vue 的 `onMounted` 会在子组件之前执行，所以是安全的。

同时添加 `onUnmounted` 清理键盘事件监听：

```js
onUnmounted(() => {
  document.removeEventListener('keydown', handleKeydown)
})
```
